# Design Decisions

Ключевые решения и трейд-оффы. Осознанные упрощения помечены как
"вне скоупа".

## Конфигурация

Все env-переменные валидируются на старте через `validate`-хук
`@nestjs/config` + class-validator (тот же стек, что и для DTO).
Класс EnvironmentVariables — одновременно схема и тип для
ConfigService, так что опечатка в ключе ловится компилятором.
Невалидный .env роняет приложение на старте с точной причиной.

## Схема данных

- `synchronize: false`, только миграции — детерминированный setup.
- `CREATE EXTENSION "uuid-ossp"` идёт первой строкой миграции, чтобы
  setup работал на чистом volume без ручных шагов через psql.
- users: UNIQUE(brand_id, email), не глобальный unique по email.
  Бренды — независимые базы игроков, один email может жить в
  нескольких брендах. Глобальная проверка к тому же была бы
  кросс-тенантным запросом и утекала бы фактом существования аккаунта
  в другом бренде. Следствие: логин — только парой (brandId, email).
- brand_id денормализован в sessions/raw_events — фильтр по тенанту
  без join.
- idempotency_keys: UNIQUE(scope, brand_id, key). Арбитр дубликатов —
  сама БД, а не SELECT-then-INSERT (у него окно гонки). `scope`
  (например `psp:stripe`) разводит id разных провайдеров.
- idempotency_keys.raw_event_id без FK: у ключей и событий разные
  политики retention, жёсткая связь мешала бы чистить независимо.
- sessions.revoked_at вместо DELETE — аудит-след. Сам logout-эндпоинт
  вне скоупа (по заданию нужны register/login/me).

## Структура модулей

- Entities лежат в модулях-владельцах (users/, identity/, webhooks/),
  в persistence/ — только миграции и tenant-scoped фабрика.
- Repository<T> инжектится только в модуле-владельце таблицы, остальные
  ходят через его сервис. UsersService — единственный владелец users:
  нормализация email и обработка 23505 в одном месте, tenant-scoping
  спрятан внутри.
- Владелец определяется доменом, а не схемой: Session — приватная
  деталь auth и живёт в identity, отдельный модуль только расширил бы
  доступ к таблице. raw_events/idempotency_keys принадлежат
  ingest-use-case'у.
- UserDto: whitelist-сериализация через @Expose — passwordHash не
  утечёт, даже если вернуть entity целиком. Оборотная сторона: забытый
  @Expose молча выбрасывает поле, поэтому набор полей закреплён тестом.

## Identity

- argon2id вместо bcrypt (рекомендация OWASP, memory-hard).
- Гибрид JWT + server-side session: JWT несёт sub/brandId/sessionId,
  guard проверяет подпись и живость сессии в БД → мгновенный отзыв
  через revoked_at. Чистый stateless JWT не подходит: в платёжном
  домене нужна возможность отзыва, и таблица sessions в задании
  намекает именно на это. Цена — +1 запрос в БД на авторизованный
  запрос; в продакшне закрывается кешем сессий.
- TTL сессии выводится из JWT_TTL, а не хардкодится: иначе токен,
  валидный по подписи, молча умирал бы на проверке сессии.
- Refresh — продление сессии, а не отдельный refresh-токен. Смысл
  классической пары access+refresh — отзывability при stateless
  access-токене, а здесь отзыв уже даёт сессия; второй артефакт
  дублировал бы её. POST /auth/refresh принимает ещё живой JWT,
  продлевает expires_at и выдаёт новый токен с тем же sessionId.
  Потолок — SESSION_MAX_LIFETIME (24h) от created_at, чтобы украденный
  токен нельзя было рефрешить вечно; TTL нового JWT обрезается по
  потолку. Протухший токен не обменивается — только новый логин.
- brandId в body register/login — симуляция для тестового; в проде
  определялся бы по домену бренда или API-ключу фронтенда.
- Регистрация: insert + catch 23505, без гонки конкурентных
  регистраций.
- Один ответ "Invalid credentials" на "нет юзера" и "не тот пароль" —
  анти-enumeration. Остаточный timing-канал (argon2.verify не зовётся,
  если юзера нет) известен; в проде — verify против dummy-хеша.

## Tenant isolation

- brandId живёт в AsyncLocalStorage: для JWT-маршрутов его кладёт
  interceptor из токена, для вебхуков — контроллер после проверки
  подписи. Код ниже не знает источника. ALS вместо request-scoped
  providers — без пересоздания DI-графа на каждый запрос.
- Контекст открывается middleware-ом пустым на весь запрос, interceptor
  наполняет уже открытый store мутацией. Вариант с als.run вокруг
  next.handle() в interceptor-е не работает: handler выполняется при
  подписке на Observable уже вне als.run. Тот же паттерн, что у
  nestjs-pino и nestjs-cls.
- Fail-closed: обращение к brandId вне контекста — сразу 500, а не
  тихий undefined (undefined означал бы исчезновение WHERE-условия и
  молчаливый leakage).
- TenantScopedRepositoryFactory.findOneScoped подставляет brandId из
  контекста поверх любого where — фильтр нельзя ни забыть, ни
  перезаписать. Один метод под реальную потребность; списочные и
  create-варианты добавятся с первым потребителем.
- /profile/me читает юзера через scoped-путь намеренно: хотя бы один
  JWT-маршрут должен прогонять всю цепочку middleware → interceptor →
  ALS, иначе сломанный interceptor не виден тестам.
- Чужой ресурс — 404, не 403: 403 подтверждал бы его существование.

## Webhooks

- HMAC-SHA256 от сырых байтов тела (rawBody), constant-time сравнение.
  Без валидной подписи ничего не сохраняется.
- Подписывается `<brandId>.<rawBody>`, не только тело: иначе
  перехваченную валидную пару (body, signature) можно переслать с
  другим X-Brand-Id, и событие ляжет в чужой бренд как новое.
  Проверено e2e-тестом replay-сценария.
- X-Brand-Id и :provider валидируются тем же паттерном
  `^[a-zA-Z0-9_-]{1,64}$`, что и brandId в auth: значение длиннее
  varchar(64) дало бы 500 на INSERT вместо честного 4xx.
- Идемпотентность: INSERT в idempotency_keys ... ON CONFLICT DO NOTHING
  в одной транзакции с raw_event. 0 вставленных строк = дубликат. Сбой
  после ключа откатывает и ключ — ретрай не блокируется сиротой.
- Дедупликация по externalEventId, не по хешу payload: провайдер может
  перегенерировать payload при ретрае, id держит стабильным.
- Дубликат → 200, не 409: не-2xx заставил бы провайдера ретраить
  бесконечно. Тело различает accepted/duplicate.
- Контракт payload (eventId/type/data) — допущение, в задании не задан.
  Неизвестные top-level поля принимаются, в raw_events сохраняется
  полный оригинальный payload (req.body), не обрезанный DTO. Поэтому же
  глобальный ValidationPipe работает с whitelist, но без
  forbidNonWhitelisted: новое поле в вебхуке не должно давать 400 и
  бесконечные ретраи.
- Никаких обновлений баланса в обработчике вебхука ("no direct balance
  updates" из задания): он только сохраняет сырое событие и отвечает
  200. Начисления — работа будущего отдельного процесса, который будет
  читать raw_events со status='received', проводить их в ledger и
  помечать processed. Поэтому в схеме уже есть status/processed_at,
  хотя самого процессора ещё нет. Причина разделения: вебхук ретраится
  и должен отвечать быстро, а движение денег требует проверок и не
  должно теряться при сбое — сырое событие остаётся источником правды,
  из которого всё можно пересчитать.

## Observability и ошибки

- nestjs-pino, JSON-логи. Correlation id берётся из X-Correlation-Id
  или генерируется, возвращается в хедере ответа и есть в каждой
  лог-строке и в теле ошибки.
- redact на password/authorization/cookie.
- Глобальный exception filter: `{ statusCode, code, message,
  correlationId }`, stack trace только в логах, 4xx = warn, 5xx = error.

## Rate limiting

- login 5/мин, register 10/мин, глобальный дефолт 100/мин.
- Вебхуки из тротлинга исключены: 429 на легитимный retry-всплеск PSP —
  это отложенные платёжные события. Их защита — подпись +
  идемпотентность; анти-DoS — задача gateway/LB.

## Тестирование

- Unit: IngestCallbackUseCase — ветвление accepted/duplicate,
  fail-closed контекста.
- Integration на живой БД: 5 параллельных одинаковых колбеков → ровно
  один accepted и одна запись. Мокать базу тут нельзя — тестируется
  поведение самого Postgres (constraint + транзакция).
- Tenant leakage: один email в двух брендах — независимые аккаунты;
  креденшлы brandA не работают в brandB; scoped-репозиторий не отдаёт
  чужую запись даже по точному id.
- E2e гоняются в отдельной базе `<DB_NAME>_test` (создаётся и
  мигрируется автоматически), между тестами TRUNCATE; truncateAll
  отказывается чистить базу без суффикса `_test`. Testcontainers не
  стал брать: для объёма MVP не окупается. E2e — один worker.
